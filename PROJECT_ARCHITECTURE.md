# DarkVector — Project Architecture

This document describes the design, subsystems, and operational workflows of DarkVector.

---

## 1. Subsystem Analysis & Rationale

DarkVector is built to address the "opacity problem" in typical security platforms. Every subsystem is chosen for clear technical reasons and is designed to run locally or in small, scalable micro-tiers.

### Machine Learning & Explainability Pipeline
* **Why it exists**: Security alerts often trigger based on raw anomalies (e.g. unexpected destination port usage, spikes in packet counts). We need a model that can run fast inference locally to grade anomaly likelihood without heavy GPU workloads.
* **How it works**:
  * **Feature Mapping**: On ingestion, alerts are mapped into a 41-column schema representing connection records from the classic KDD Cup 99 dataset.
  * **Isolation Forest**: We use `scikit-learn`'s `IsolationForest` implementation. It isolates observations by randomly selecting a feature and a split value. Outliers require fewer splits to isolate, resulting in shorter paths in the trees.
  * **SHAP Explainability**: We load the trained forest into SHAP's `TreeExplainer` to calculate the exact marginal contributions of each feature to the model's anomaly decision. The top 5 features with the highest absolute Shapley values are extracted as explanation factors.
  * **Risk Calibration**: The raw anomaly score is mapped into a discrete percentile-based risk rating (0-100) using the score distribution from the training run.
* **Where it fits**: It runs in the backend service layer on alert ingestion and commits its results directly to the database in `analysis_json`.

### Behavioral Detection Engine
* **Why it exists**: While machine learning models are excellent at finding anomalous statistical behavior, they fail at recognizing known, deterministic attacker tradecraft (e.g., executing a command prompt from Excel). The behavioral detection engine handles pattern-based threat identification.
* **How it works**:
  * **Process Tree Construction**: Raw `process_start` events are parsed and aggregated into a hierarchical tree based on `process_guid` and `parent_process_guid`.
  * **Incremental Scheduler**: The `DetectionScheduler` runs regularly (e.g. triggered by telemetry influx), keeping a timestamp cursor of processed events and storing rolling process start context in memory to link trees across scheduler runs.
  * **Rule Registry**: An extensible registry of `DetectionRule` classes. Each rule implements an `evaluate(process_tree)` method returning structured detections.
* **Where it fits**: Sits in `app/services/detection/` and executes incrementally, creating correlated investigations in the database.

### AI Context Engine (Vector AI)
* **Why it exists**: Analysts spend valuable time translating raw JSON dumps (SHAP values, alert details, host vitals) into human-readable narratives. The AI Context Engine provides this translation safely.
* **How it works**:
  * **Context Aggregation**: The `ContextBuilder` aggregates investigation details, memory logs, and local RAG resources.
  * **Briefing Compilation**: `KnowledgePackV2` converts this context into a natural-language brief.
  * **Route Optimization**: An `IntentClassifier` runs query strings through keyword checks to route questions to specialized builders (`General`, `ExplainAttack`, `Risk`, `Remediation`, `Mitre`, `Timeline`, `Evidence`), avoiding LLM routing lag.
  * **Evidence Mapping**: Deterministic citations are appended directly from the database schema to verify the LLM's references.
  * **Output Validation**: `ResponseValidatorV2` inspects the response to ensure it maps correct techniques and contains no generic placeholder text. A fallback deterministic provider runs if the validation fails.
* **Where it fits**: Integrates in `app/services/llm/` and hooks into the `/chat` and `/report` API endpoints.

### Telemetry Ingestion Pipeline
* **Why it exists**: An security dashboard is useless without live data. The telemetry pipeline acts as the system's receiver.
* **How it works**:
  * **DV Sentinel Daemon**: Runs a background schedule on client endpoints, fetching CPU, memory, OS properties, and uptime via `psutil`.
  * **APITransport**: Sends vitals over HTTPS to the FastAPI server, backed by exponential retry logic.
  * **Telemetry Bus**: Validates API keys, writes raw events to `telemetry_events`, and updates `endpoint_agents` metadata.
* **Where it fits**: Defined under `sentinel/` (agent side) and `app/services/telemetry/` (backend side).

## 2. Database Schema

### `telemetry_events` Table
Tracks every event received via the Telemetry Bus.

* `id` (VARCHAR, PK): UUID4 string.
* `host_id` (VARCHAR, Indexed): Stable identifier generated by the agent.
* `hostname` (VARCHAR): Name of the client host.
* `timestamp` (DATETIME): Event generation timestamp.
* `event_type` (VARCHAR): Type of event (e.g., `heartbeat`, `process_start`).
* `severity` (VARCHAR): Event severity.
* `source` (VARCHAR): Ingestion agent identifier.
* `payload` (JSON): Arbitrary raw telemetry attributes.
* `created_at` (DATETIME): Server ingestion timestamp.

### `endpoint_agents` Table
Maintains the active inventory and connection status of endpoints.

* `id` (VARCHAR, PK): UUID4 string.
* `host_id` (VARCHAR, Unique, Indexed): Stable host hardware identifier.
* `hostname` (VARCHAR): Reported hostname.
* `agent_version` (VARCHAR): Endpoint agent code version.
* `os` (VARCHAR): Operating system family (Windows, Linux, Darwin).
* `architecture` (VARCHAR): CPU architecture.
* `ip_address` (VARCHAR): Last reported public/local IP.
* `status` (VARCHAR): Agent status (`connected`, `offline`).
* `last_seen` (DATETIME): Timestamp of last received heartbeat.
* `created_at` (DATETIME): Registration timestamp.
* `updated_at` (DATETIME): Time of last state change.

### `investigations` Table
Tracks security investigations.

* `investigation_id` (VARCHAR, PK): Unique case ID.
* `title` (VARCHAR): Analyst-facing case title.
* `status` (VARCHAR): Case lifecycle state (`NEW`, `INVESTIGATING`, `CONTAINED`, `RESOLVED`).
* `severity` (VARCHAR): Case severity.
* `risk_score` (FLOAT): Anomaly score metric.
* `assigned_analyst` (VARCHAR): Target assignee.
* `alert_json` (JSON): Copy of the original triggering alert parameters.
* `analysis_json` (JSON): Shapley values, risk percentile, and MITRE mapping results.
* `containment_status` (VARCHAR): Status of isolation job (`QUEUED`, `EXECUTING`, `COMPLETED`, `FAILED`).
* `containment_message` (TEXT): Status detail.
* `created_at` (DATETIME): Case creation timestamp.
* `updated_at` (DATETIME): Case modification timestamp.

## 3. Operations & Workflows

### Telemetry Heartbeat Ingestion
```
[DV Sentinel Agent]
       │ Collects CPU%, Mem%, OS details, primary IP, and uptime
       ▼
[APITransport HTTP Client]
       │ Serializes package to TelemetryEvent DTO
       │ POSTs to FastAPI /api/v1/telemetry with Bearer Key
       ▼
[FastAPI Telemetry Router]
       │ Validates X-API-Key header
       │ Hands payload to TelemetryIngestionService
       ▼
[Telemetry Bus]
       ├─ Writes to telemetry_events table
       └─ Calls EndpointRepository.create_or_update()
             ├─ Updates endpoint_agents properties
             └─ Marks status as "connected" and sets last_seen
```

### Anomaly Analysis & Investigation Workflow
```
[Alert Ingestion API]
       │ Triggered via manual test or Sentinel telemetry events
       ▼
[InferenceService.analyze()]
       ├─ FeatureMapper: Converts raw payload variables to KDD 41 features
       ├─ Preprocessor: Normalizes categorical/scaler variables
       ├─ Isolation Forest: Calculates anomaly distance score
       ├─ SHAP Explainer: Generates marginal feature contributions
       └─ ContextService: Looks up MITRE keywords and IP reputation
       ▼
[InvestigationService.create_from_analysis()]
       ├─ Persists target in investigations table
       └─ Auto-logs "alert_created" and "analysis_completed" timeline events
       ▼
[Analyst Frontend Workspace]
       ├─ Dashboard updates, displaying the priority investigation card
       ├─ Workspace loads process tree and active timeline logs
       └─ AI Analyst Chat loads Context and opens specialized router
```
