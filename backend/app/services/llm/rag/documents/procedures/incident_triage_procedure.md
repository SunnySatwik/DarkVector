# Incident Triage Procedure

## Purpose

This procedure defines the standard steps for triaging security alerts upon ingestion into the SOC queue. All Tier 1 analysts must follow this procedure before escalating, closing, or investigating an alert.

## Scope

Applies to all alerts generated by:
- SIEM (Splunk, Sentinel, QRadar)
- EDR (CrowdStrike, SentinelOne, Defender)
- Network IDS/IPS
- DarkVector AI Platform

## SLA Targets

| Severity | Initial Triage | Escalation Deadline |
|----------|---------------|---------------------|
| CRITICAL | 5 minutes | 15 minutes |
| HIGH | 15 minutes | 1 hour |
| MEDIUM | 1 hour | 4 hours |
| LOW | 4 hours | Next business day |

## Step 1 — Initial Classification (0–5 min)

Upon alert receipt, immediately determine:

1. **Alert source** — Which tool generated the alert?
2. **Asset criticality** — Is the affected host a domain controller, database server, endpoint, or workstation?
3. **Account type** — Is the involved account a service account, privileged admin, or standard user?
4. **Environment** — Is this production, staging, or development?

Use the above factors to assign an initial severity score. A domain controller + privileged account + production environment = automatic CRITICAL classification.

## Step 2 — Context Enrichment (5–15 min)

Pull supporting context for the alert:

- [ ] Threat Intelligence lookup on IPs, domains, and file hashes
- [ ] Historical alerts for the same source host (last 72h)
- [ ] Historical alerts for the same user account (last 30 days)
- [ ] Geolocation and ASN check on external IPs
- [ ] WHOIS and passive DNS for suspicious domains
- [ ] VirusTotal, AbuseIPDB, or MISP correlation

## Step 3 — True Positive / False Positive Decision

Based on enrichment results, classify the alert:

| Classification | Action |
|----------------|--------|
| True Positive | Escalate to investigation per SLA |
| Likely True Positive | Begin investigation; escalate if confirmed |
| False Positive | Document reason; close; consider tuning |
| Benign | Close with note; no action required |
| Undetermined | Escalate to Tier 2 with all enrichment data |

## Step 4 — Documentation

Before escalating or closing, record in the ticket:

1. Alert summary (what happened, on which host, by which account)
2. Enrichment results
3. Classification rationale
4. Initial severity assignment
5. Actions taken (if any)

## Step 5 — Escalation

If escalating to Tier 2 or Incident Response:

1. Update the ticket status to **Escalated**.
2. Assign to the on-call Tier 2 analyst.
3. Post a brief in the #soc-incidents Slack channel (CRITICAL and HIGH only).
4. If CRITICAL: page the on-call IR lead directly.

## Common False Positive Patterns

| Alert Type | Common Cause |
|------------|-------------|
| Antivirus alert on PDF | Macro scanning, benign |
| Admin login from new IP | VPN exit node rotation |
| Multiple failed logins | Password manager sync issue |
| Lateral movement to DC | Legitimate admin tooling |
| PowerShell encoded command | Group Policy update script |

Always cross-reference with change management tickets before closing as false positive.
